# Phase 4: Analytics, Polish & Launch - Research

**Researched:** 2026-05-14
**Domain:** Cloudflare Workers Static Assets headers + Umami Cloud analytics + DNS cutover + Lighthouse production audit
**Confidence:** HIGH on Cloudflare `_headers` syntax, Lighthouse CLI, DNS cutover; MEDIUM on Umami events endpoint origin (history of churn — verify empirically); LOW on Twitter Card Validator official replacement (deprecated, no first-party replacement).

## Summary

Phase 4 is a wiring phase, not a building phase. Every piece (analytics snippet, headers file, DNS rules) is a small, well-documented Cloudflare or Umami primitive — the risk is in sequencing and verification, not invention. The only research-grade finding that should change the plan: the Umami Cloud events endpoint origin has shifted three times in the last two years (analytics.umami.is → api-gateway-eu.umami.dev → api-gateway.umami.dev → and now apparently `cloud.umami.is` itself per the most recent CSP discussions), so the CSP `connect-src` for Umami must be **derived empirically from a network-tab inspection on the preview deploy** (D-02's pre-cutover verification step), not hard-coded from a 12-month-old doc. Build the CSP "open enough to ship, tight enough to be useful" — `https:` on connect-src for the umami.is/.dev family is acceptable v1.

The `_headers` file works the way Cloudflare Pages had it work: drop a `_headers` file in `public/`, the build copies it to `dist/`, Workers Static Assets serves it. The 2024 Astro issue #13164 (Beta Assets, `_headers` not honored) was filed against the *Beta* iteration; current Cloudflare official docs document `_headers` as the supported, recommended mechanism, including the exact `/static/*  Cache-Control: public, max-age=31556952, immutable` pattern we want for `/_astro/*`. `_headers` does NOT apply to responses generated by Worker code, but per Phase 3 D-22 the only Worker-side path reserved is `/api/*` — and nothing ships there in v1. So `_headers` is the right tool.

**Primary recommendation:** Single small wave. Plan = (1) Umami snippet + 4 `data-umami-event` attributes; (2) `public/_headers` with HSTS + CSP + Permissions-Policy + cache-control; (3) `SETUP-DNS.md` at repo root; (4) launch-day execution writing `LAUNCH-REPORT.md`. Use Lighthouse CLI v13.1 (already installed) — not PageSpeed Insights API — for production audits because we can re-run it without rate limits and it matches the Phase 1 CI baseline.

### Pattern 1: `_headers` file (Cloudflare Workers Static Assets)

**What:** A plain-text file at `public/_headers` that Astro copies to `dist/_headers`. Cloudflare Workers Static Assets reads it and applies the specified headers per path.

**When to use:** Every static response from this site. Worker-side responses (`/api/*` per `run_worker_first`) would NOT receive these headers — but no `/api/*` route ships in v1.

**Syntax (verbatim from Cloudflare docs):**
```
/path                   ← URL pattern (one per block, top of block, no leading space)
  Header-Name: value    ← indented two spaces, one header per line
  Another-Header: value
                        ← blank line ends the block
/other-path
  Header-Name: value
```

**Wildcards:**
- `/static/*` — splat (greedy)
- `/movies/:title` — placeholder (single segment), with `:title` usable in header values

**Source:** [developers.cloudflare.com/workers/static-assets/headers/](https://developers.cloudflare.com/workers/static-assets/headers/) `[VERIFIED]`

### Pattern 2: Env-aware Umami `<script>` (D-02)

**What:** A `<script async>` tag in `BaseLayout.astro` with `data-domains` computed at build time from the same `isProduction()` helper that `robots.txt.ts` already uses.

**When to use:** Every page. BaseLayout's `<head>` is the single insertion point — every page composes through BaseLayout (verified by reading `src/pages/*.astro` — all 6 prerendered surfaces use `<BaseLayout>`).

**Example:**
```astro
---
// src/layouts/BaseLayout.astro (excerpt — actual placement: after <Font/> tags, just before <slot name="head" />)
import { isProduction } from '../lib/site-url';

const UMAMI_ID = import.meta.env.PUBLIC_UMAMI_WEBSITE_ID;
const APEX = 'studiobluemli.com';

// In production: apex only.
// Otherwise: apex + workers.dev preview (so pre-cutover smoke tests in D-02 verification appear in Realtime).
// The workers.dev hostname is read from CF_WORKERS_URL / PUBLIC_CF_WORKERS_URL (same vars resolveAssetBase() reads).
const isProd = isProduction();
const previewHostname = !isProd
  ? (() => {
      const raw = process.env?.CF_WORKERS_URL
        ?? import.meta.env.PUBLIC_CF_WORKERS_URL
        ?? '';
      try { return raw ? new URL(raw).hostname : ''; } catch { return ''; }
    })()
  : '';
const dataDomains = isProd
  ? APEX
  : [APEX, previewHostname].filter(Boolean).join(',');
---
{UMAMI_ID && (
  <script
    async
    src="https://cloud.umami.is/script.js"
    data-website-id={UMAMI_ID}
    data-domains={dataDomains}
  />
)}
```

**Why this shape:**
- `{UMAMI_ID && (...)}` — local dev with no env var set silently emits no snippet (instead of an undefined-id script that 4xxs forever). Mirrors `robots.txt.ts`'s safe-default-Disallow stance.
- `async`, not `defer` — Umami's official docs use `async`; defer is fine but matches less of the documentation, raising review questions.
- No `data-host-url` — Umami Cloud uses its default endpoint when the script is served from `cloud.umami.is`. Override only needed for self-hosted (per docs.umami.is).
- No `data-auto-track="false"` — we WANT auto-tracking (pageviews). Custom events compose with auto-tracking, they don't replace it.

### Pattern 3: `data-umami-event` attributes on existing elements

**What:** Per docs.umami.is/track-events, adding `data-umami-event="<slug>"` to any element makes Umami record the event on interaction (click for `<a>`/`<button>`). No additional JS, no `client:` directive.

**When to use:** The 4 elements identified in D-01.

**Example (gallery card):**
```jsx
// src/components/design-skill/GalleryGrid.jsx — already renders <a class="card" href={`/gallery/${slug}`}>
// Edit: add data-umami-event attribute to the wrapping <a>.
<a class="card" href={`/gallery/${slug}`} data-umami-event="gallery_card_click">
  …
</a>
```

**Anti-Patterns to Avoid:**
- **Adding `client:load` to attach a click handler manually:** Umami's auto-handling is sufficient and ships zero extra JS. The Phase 1 contract is "no client-side React"; that survives.
- **Putting the website ID in `site.yaml`:** Works, but creates a CNT-NN schema change and forces a content-collection re-read for what is operational config (not content). Use env var.
- **Inlining the Umami snippet directly into `index.astro` only:** BaseLayout is the single insertion point per Phase 1 contract; per-page would mean 5 places to keep in sync.

## Common Pitfalls

### Pitfall 1: Umami events endpoint origin drifts and breaks CSP

**What goes wrong:** A tight `connect-src cloud.umami.is` CSP allows the script to load but blocks event POSTs because Umami's tracker POSTs to a *different* origin (historically: `analytics.umami.is`, then `api-gateway-eu.umami.dev`, then `api-gateway.umami.dev`, possibly now back to `cloud.umami.is/api/send`). Events 0. Founder sees no Realtime data after 5 min. Looks like Pitfall #19 (domain not registered in Umami) but the actual cause is a CSP block visible only in DevTools Console.

**Why it happens:** The Umami events endpoint is not formally stable across the script's history. GitHub Discussion #2719 (Aug 2024 onward) documents the rolling change with no authoritative consolidated answer.

**How to avoid:**
1. Ship CSP with `connect-src 'self' https://cloud.umami.is https://*.umami.is https://*.umami.dev` — the wildcard covers the family of endpoints Umami has used.
2. **Empirical verification step** in D-04 launch-checklist item 5: when clicking each of the 4 events, the founder/Claude opens DevTools → Network and confirms an outgoing `POST` with status 200/204 to *some* umami origin. If blocked by CSP, the Console shows `Refused to connect to 'https://…' because it violates the following Content Security Policy directive`. Fix = add the origin to `_headers`.
3. If a permission to widen CSP is needed under pressure, the safe interim is `connect-src 'self' https:` (any HTTPS) — still better than no CSP at all.

**Warning signs:** Umami Realtime view shows 0 events 5 min after a click; DevTools Console shows a CSP `connect-src` refusal.

### Pitfall 2: `_headers` only applies to STATIC asset responses, not Worker-generated responses

**What goes wrong:** Headers set in `public/_headers` for `/api/*` paths never fire because `run_worker_first: ["/api/*"]` (per `wrangler.jsonc` line 11) sends `/api/*` to Worker code, and Cloudflare's official docs explicitly state: *"Custom headers defined in the `_headers` file are not applied to responses generated by your Worker code."*

**Why it happens:** Workers Static Assets and the Worker runtime are separate response pipelines. `_headers` is parsed by Static Assets only.

**How to avoid (Phase 4 specific):** No `/api/*` route ships in v1 (Phase 3 D-22 — the `run_worker_first` config is reserved for a future contact form). The `dist/server/entry.mjs` entrypoint exists but no route routes to it. So `_headers` covers 100% of v1 traffic. Document this explicitly in the `_headers` file as a comment so a future engineer adding `/api/*` knows to also inject headers Worker-side.

`[VERIFIED: developers.cloudflare.com/workers/static-assets/headers/]`

### Pitfall 3: HSTS preload-list submission is effectively one-way

**What goes wrong:** The site ships HSTS with the `preload` directive AND someone submits to hstspreload.org. Then the founder wants to change a subdomain back to HTTP or drop apex HTTPS for a day. Modern browsers cache the preload entry for ~1 year and refuse to connect over HTTP. Effectively un-rollback-able.

**Why it happens:** Browser vendors bake the preload list into shipping browser versions, on a multi-month cadence. Removing a domain is a long process.

**How to avoid:**
- Phase 4 SHIPS the HSTS header WITH `preload` directive (LCH-04, D-09) but does NOT submit to hstspreload.org.
- Document this in `SETUP-DNS.md` as "deferred to v1.x after 30 days clean."
- Confirm DNS for `www.studiobluemli.com` exists (it must, for the 301 to be reachable) and that `www.studiobluemli.com` serves HTTPS (the Redirect Rule will serve the 301 *over* HTTPS — Cloudflare-issued cert) — these are preload prerequisites if we ever do submit later.

`[CITED: hstspreload.org/]`

### Pitfall 5: Custom Domain DNS cutover before cert provisions = transient TLS errors

### `public/_headers` content (target shape)

```
# Cloudflare Workers Static Assets headers file (LCH-04, D-08, D-09).
# Applies to every static asset response. Does NOT apply to Worker code (v1 ships no /api/*).
#
# Order: most-specific path first, '*' / '/' last (CF applies first match per header per response,
# but every block is read; specific paths can override broad ones).
# Reference: https://developers.cloudflare.com/workers/static-assets/headers/

# --- Fingerprinted Astro assets (CSS, JS, fonts emitted by Vite/Astro build) ---
# Content-hashed → safe to cache forever.
/_astro/*
  Cache-Control: public, max-age=31536000, immutable

# --- Static images with stable filenames ---
# 7 days = founder can refresh a hero on a PR-preview-then-merge loop and not wait too long for stale caches.
/gallery/*
  Cache-Control: public, max-age=604800
/og-default.png
  Cache-Control: public, max-age=604800
/mark.svg
  Cache-Control: public, max-age=604800
/favicon.svg
  Cache-Control: public, max-age=604800
/favicon-16.png
  Cache-Control: public, max-age=604800
/favicon-32.png
  Cache-Control: public, max-age=604800
/favicon.ico
  Cache-Control: public, max-age=604800
/apple-touch-icon.png
  Cache-Control: public, max-age=604800

# --- Everything else (HTML, robots.txt, sitemap-*.xml) — always fresh ---
# Cloudflare's edge layer caches at its own discretion; we just tell browsers not to.
/*
  Cache-Control: public, max-age=0, must-revalidate
  Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
  X-Content-Type-Options: nosniff
  Referrer-Policy: strict-origin-when-cross-origin
  Permissions-Policy: accelerometer=(), camera=(), geolocation=(), gyroscope=(), magnetometer=(), microphone=(), payment=(), usb=(), interest-cohort=(), browsing-topics=()
  Content-Security-Policy: default-src 'self'; script-src 'self' https://cloud.umami.is; connect-src 'self' https://cloud.umami.is https://*.umami.is https://*.umami.dev; img-src 'self' data:; style-src 'self' 'unsafe-inline'; font-src 'self' data:; frame-ancestors 'none'; base-uri 'self'; form-action 'self'
```

**Notes on the CSP composition:**
- `default-src 'self'` — deny-by-default for any directive not explicitly set.
- `script-src 'self' https://cloud.umami.is` — Umami's `<script src="https://cloud.umami.is/script.js">` plus any first-party scripts. No `'unsafe-eval'`. No `'unsafe-inline'` on scripts.
- `connect-src 'self' https://cloud.umami.is https://*.umami.is https://*.umami.dev` — covers the events POST. The wildcards mitigate Pitfall 1 (endpoint origin drift).
- `img-src 'self' data:` — first-party images; `data:` allows tiny SVG/PNG data-URIs (e.g., Mark.jsx inline shapes if any). Per Phase 3 there are no third-party images.
- `style-src 'self' 'unsafe-inline'` — Astro emits scoped `<style>` blocks AND React JSX components emit `style={{...}}` inline attributes. `'unsafe-inline'` is required for both. No external stylesheet origins needed (Astro emits to `/_astro/*` which matches `'self'`).
- `font-src 'self' data:` — Fontsource WOFF2s are first-party. `data:` covers any embedded fallback.
- `frame-ancestors 'none'` — no embedding allowed (sets X-Frame-Options behavior in modern browsers).
- `base-uri 'self'` and `form-action 'self'` — defense-in-depth against `<base>` injection and form-submission hijacking.

**Permissions-Policy notes:**
- `interest-cohort=()` disables FLoC (Chrome's now-retired ad signal but the header is still emitted defensively).
- `browsing-topics=()` disables FLoC's successor.
- Camera/mic/geo/payment/USB/accelerometer/gyroscope/magnetometer covered per D-09.
- `[CITED: w3.org/TR/permissions-policy/]` for header structure (empty allowlist disables for all origins including same-origin).

**Cache strategy rationale:**
- `/_astro/*` is content-hashed by Vite → safe `immutable` + 1y.
- Static images: 7d picked. 1d would mean a refresh-after-merge would clear within a day for return visitors; 7d means it might take a week. Tradeoff: 7d wins on CDN hit-rate (most relevant for IG-DM-driven traffic landing once and forgetting) and reflects that gallery photos rarely change (Phase 2 contract — pieces don't get re-photographed, they get a new slug instead).
- HTML: `max-age=0, must-revalidate` is the standard "validate every visit" pattern; browsers send `If-None-Match` and Cloudflare returns 304 — fast and always-current.

### OG-image HEAD check (D-04 item 3)

The cleanest path is to walk the production `sitemap-0.xml` (which Phase 3 confirmed has 11 entries: 5 top-level + 6 gallery slugs), fetch each page, extract the `og:image` URL, and HEAD-check it.

```bash
#!/usr/bin/env bash
# scripts/check-og-images.sh — D-04 item 3.
# Walks every URL in sitemap-0.xml on production, extracts og:image, HEAD-checks each.
set -euo pipefail

SITE="https://studiobluemli.com"
SITEMAP="$SITE/sitemap-0.xml"

# Extract URLs from sitemap (one per line).
URLS=$(curl -fsS "$SITEMAP" | grep -oE '<loc>[^<]+</loc>' | sed -E 's|</?loc>||g')

FAIL=0
for URL in $URLS; do
  # Extract og:image from the page.
  OG=$(curl -fsS "$URL" | grep -oE '<meta property="og:image" content="[^"]+"' | head -1 | sed -E 's|.*content="([^"]+)".*|\1|')
  if [ -z "$OG" ]; then
    echo "FAIL: $URL has no og:image"
    FAIL=1
    continue
  fi
  STATUS=$(curl -fsS -o /dev/null -w '%{http_code}' -I "$OG")
  if [ "$STATUS" != "200" ]; then
    echo "FAIL: $URL → og:image $OG → HTTP $STATUS"
    FAIL=1
  else
    echo "OK: $URL → $OG (200)"
  fi
done

if [ "$FAIL" -eq 1 ]; then exit 1; fi
echo "All og:image URLs return 200."
```

Bash one-liner is workable but multi-line script is clearer for `LAUNCH-REPORT.md` evidence. Save as `scripts/check-og-images.sh` if reuse is wanted; otherwise inline once in the launch-day execution. Recommend the script form — same pattern as Phase 3's `scripts/check-seo-output.mjs`.

### Known Threat Patterns for Static + Cloudflare + Astro

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Clickjacking via iframe embedding | Tampering | `Content-Security-Policy: …; frame-ancestors 'none'` in `_headers` |
| MIME-sniffing attacks (browser interprets `.png` as `.html`) | Spoofing | `X-Content-Type-Options: nosniff` |
| Referrer leakage to third parties (Instagram clicks) | Information Disclosure | `Referrer-Policy: strict-origin-when-cross-origin` |
| Powerful API misuse (camera, mic, geolocation) | Tampering | `Permissions-Policy: camera=(), …` |
| Protocol downgrade (HTTPS → HTTP MITM) | Tampering | `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload` |
| XSS via inline event handlers or eval | Tampering | CSP `script-src 'self' https://cloud.umami.is` (no `'unsafe-eval'`, no `'unsafe-inline'`) |
| Third-party-script supply-chain compromise | Tampering | Only ONE third-party script (`cloud.umami.is/script.js`); CSP `script-src` restricts to that origin. SRI deferred (v1.x) because Umami script URL is versionless. |
| FLoC/Topics fingerprinting | Information Disclosure | `Permissions-Policy: interest-cohort=(), browsing-topics=()` |

Phase 4's security posture is "best-effort for a marketing-only static site": no auth, no PII collection, no API, no JS runtime hostility surface. The headers above are the standard Cloudflare-recommended set for a v1 marketing site.
